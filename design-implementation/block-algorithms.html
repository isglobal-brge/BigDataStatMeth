Block Algorithms

Purpose

This section explains the design and implementation of block-wise algorithms in BigDataStatMeth, which enable computations on datasets that are too large to fit entirely into memory. The same logic applies both in-memory and for HDF5-based operations.

Why block algorithms?

Working with large matrices often exceeds the available RAM. To solve this, BigDataStatMeth divides matrices into smaller blocks that can be loaded, processed, and written back independently.

This approach:

  • keeps memory usage under control,
  • enables parallel processing across multiple threads, and
  • allows scalability to terabyte-scale data.

Conceptual view

Implementation overview

The block-wise approach is implemented in both HDF5 and in-memory modules.

Context Function example Description
HDF5 bdblockmult_hdf5() Multiplies two matrices stored in HDF5 files by blocks.
Memory bdblockMult() Performs block multiplication on matrices fully in RAM.

In both cases, the algorithm iterates through blocks of fixed size and performs local operations (multiplication, cross-product, QR, SVD, etc.).

Pseudocode example

TipPseudocode 
for each block_i in MatrixA:
    for each block_j in MatrixB:
        read block_i and block_j into memory
        compute partial_result = block_i × block_j
        write partial_result to output file
Note

Notes The block size determines the trade-off between memory use and I/O speed.

The number of threads controls parallelism across blocks.

If threads > available blocks, some cores remain idle.

Internal mechanism

The workflow for HDF5 operations generally follows these steps:

  1. Initialize datasets
    • Open HDF5 files and identify matrix dimensions.
    • Allocate output datasets if they do not exist.
  2. Define block structure
    • Split matrix dimensions into chunks (block_size).
    • Each block is identified by its indices in rows and columns.
  3. Iterate and process
    • For each block: read → compute → write.
    • Intermediate results are directly streamed to disk to avoid RAM overflow.
  4. Synchronization
    • When threads > 1, blocks are processed in parallel.
    • Synchronization ensures each thread writes to different disk regions.
  5. Finalize
    • Close datasets and flush HDF5 buffers.

Performance considerations

  • Block size
    Affects the balance between computation and I/O. Optimal values often range from 1024 to 4096 rows/columns depending on system memory and disk speed.

  • Threads
    The parameter threads defines the number of concurrent tasks. Excessive threads can cause contention in disk writes.

  • Compression
    HDF5 datasets are stored by default with compression level = 6. Higher compression reduces file size but increases CPU time.

Warning

When using multiple threads, concurrent writes to HDF5 can become a bottleneck. It is recommended to adjust block size and thread count according to available resources.

Summary

  • BigDataStatMeth uses block-wise algorithms to process large datasets efficiently.
  • Computation proceeds block by block, minimizing memory usage.
  • Parallelism and compression further improve scalability.
  • The same design principles apply in both in-memory and HDF5-based contexts.
