HDF5 Performance Optimization

Chunking, Compression, and Efficient Processing

Quick Overview

Noteüìã TL;DR - Performance Essentials

Three pillars of HDF5 performance:

  1. Chunking: Determines I/O efficiency
  2. Compression: Reduces file size and I/O time
  3. Access patterns: Match operations to data layout

Quick wins: - Use chunk size 10 KB - 1 MB - Enable compression (level 4-6 for balance) - Use BigDataStatMeth‚Äôs automatic block sizing - Enable parallel processing (paral=TRUE)

Performance gains: - Good chunking: 10-100x faster I/O - Compression: 3-10x smaller files - Block processing: Unlimited data size - Parallelization: Near-linear speedup

Conceptual Understanding

The Performance Triangle

HDF5 performance depends on three interrelated factors:

        Chunking
          /  \
         /    \
        /      \
  Compression - Access Pattern

BigDataStatMeth optimizes all three automatically

What is Chunking?

Think of a matrix as a bookshelf:

Without chunking (contiguous):

[Book spanning entire shelf]
- Must move entire book to read one page
- Cannot add pages once placed

With chunking:

[Many smaller books on shelf]
- Read only the book you need
- Can rearrange or add books
- Access any section quickly

HDF5 chunks are independent blocks:

Matrix [1000 √ó 1000]
‚îÇ
‚îú‚îÄ Chunk [100 √ó 100]  ‚Üê 100 chunks total
‚îú‚îÄ Chunk [100 √ó 100]
‚îú‚îÄ Chunk [100 √ó 100]
‚îî‚îÄ ...

Why Chunking Matters

Example: Reading columns

Bad chunking (row-oriented):

Chunks:  [Row 1-100, All Cols]
         [Row 101-200, All Cols]
         
To read column 1:
‚Üí Read ALL chunks (100% of file!)

Good chunking (square blocks):

Chunks:  [100√ó100] blocks
         
To read column 1:
‚Üí Read only 10 chunks (10% of file)

Performance difference: 10x faster!

Compression Benefits

Compression provides two benefits:

1. Smaller files (obvious)

Uncompressed:  1 GB
Compressed:    200 MB  (5x smaller)

2. Faster I/O (surprising!)

Read 1 GB from disk:     100 ms
Read 200 MB + decompress: 40 ms + 20 ms = 60 ms

Result: 40% faster!

Why? Disk I/O is slower than CPU decompression.

BigDataStatMeth‚Äôs Approach

BigDataStatMeth handles performance optimization automatically:

What it does: 1. Analyzes dataset dimensions 2. Calculates optimal block size 3. Aligns blocks with HDF5 chunks 4. Processes data in efficient blocks 5. Parallelizes when beneficial

You control: - blocksize parameter (optional) - paral flag (enable/disable parallelization) - threads count (number of parallel workers)

Technical Details

Tipüî¢ Performance Specifications (click to expand)

Chunk Size Guidelines

Optimal chunk size: 10 KB to 1 MB

Too small (< 1 KB): - High overhead (many chunk lookups) - Inefficient caching - Slow iteration

Too large (> 10 MB): - Defeats partial I/O purpose - Memory pressure - Slow single-element access

Formula for chunk size:

chunk_size = chunk_dims[1] √ó chunk_dims[2] √ó element_size

Example:
chunk = [100, 100]
element_size = 8 bytes (double)
chunk_size = 100 √ó 100 √ó 8 = 80 KB  ‚úì Good!

Block Size Selection

BigDataStatMeth uses intelligent block sizing:

Factors considered: 1. Available memory: Avoid swapping 2. Dataset dimensions: Match to data shape 3. Chunk layout: Align with chunks 4. Cache size: Maximize hit ratio 5. Operation type: Different for PCA vs multiplication

Automatic selection:

# BigDataStatMeth chooses automatically
bdPCA_hdf5(file, group, dataset)

# Typical selections:
# Small data (< 1 GB):     Load all
# Medium data (1-10 GB):   blocksize ~ 1000
# Large data (> 10 GB):    blocksize ~ 256-512

Manual override:

# Advanced users can override
bdblockmult_hdf5(file, group, A, B,
                 blocksize = 128)

Compression Algorithms

Available compressors:

Algorithm Ratio Speed CPU Best For
None 1.0x Fastest None Pre-compressed data
GZIP 1 2-4x Fast Low Quick compression
GZIP 4-6 3-5x Medium Medium Recommended balance
GZIP 9 4-7x Slow High Archival storage

Compression effectiveness by data type:

Random floats:        1.1x  (poor)
Natural images:       2-3x  (OK)
Genomic data (SNPs):  3-5x  (good)
Sparse matrices:     10-50x (excellent)
Integer sequences:    5-10x (very good)

BigDataStatMeth default: GZIP level 6 (good balance)

Cache Performance

HDF5 chunk cache:

Default: 1 MB cache, 521 slots
Optimal: 10-50 MB cache, prime number slots

Cache hit ratio:

Good:  > 90% hits
OK:    70-90% hits
Bad:   < 70% hits (increase cache or adjust chunks)

BigDataStatMeth operations are designed for high cache efficiency.

Parallel I/O Performance

Thread-level scaling:

1 thread:   100 MB/s baseline
2 threads:  180 MB/s (90% efficiency)
4 threads:  320 MB/s (80% efficiency)
8 threads:  560 MB/s (70% efficiency)

Scaling factors: - Dataset size (larger = better scaling) - Operation complexity - I/O vs CPU ratio - Memory bandwidth

I/O Performance Model

Without compression:

Time = n_blocks √ó seek_time + (data_size / bandwidth)
     = 10 √ó 10 ms + (100 MB / 200 MB/s)
     = 100 ms + 500 ms = 600 ms

With compression:

Time = n_blocks √ó seek_time + (compressed_size / bandwidth) + decompress_time
     = 10 √ó 10 ms + (20 MB / 200 MB/s) + 50 ms
     = 100 ms + 100 ms + 50 ms = 250 ms  (2.4x faster!)

# - Aligns with HDF5 chunks

Manual Block Size Tuning

For advanced users, block size can be tuned:

# Default: automatic
result1 <- bdblockmult_hdf5(
  filename = "data.hdf5",
  group = "matrices",
  A = "matrix_a",
  B = "matrix_b"
)

# Manual tuning for specific hardware
result2 <- bdblockmult_hdf5(
  filename = "data.hdf5",
  group = "matrices",
  A = "matrix_a",
  B = "matrix_b",
  blocksize = 256  # Tune for your system
)

Guidelines for manual tuning:

# Small matrices (< 1000 √ó 1000)
blocksize = 512  # Larger blocks

# Medium matrices (1000-10000 √ó 1000-10000)
blocksize = 256  # Default, good balance

# Large matrices (> 10000 √ó 10000)
blocksize = 128  # Smaller blocks, better memory

# Very large matrices (> 100000 √ó 100000)
blocksize = 64   # Minimal memory footprint

Parallel Processing

Enable parallelization for CPU-intensive operations:

# Enable parallel processing
pca <- bdPCA_hdf5(
  filename = "data.hdf5",
  group = "data",
  dataset = "matrix",
  k = 20,
  paral = TRUE,     # Enable parallelization
  threads = 4       # Number of threads
)

# Matrix multiplication with parallelization
result <- bdblockmult_hdf5(
  filename = "data.hdf5",
  group = "matrices",
  A = "matrix_a",
  B = "matrix_b",
  paral = TRUE,
  threads = NULL    # NULL = use all available cores
)

When to use parallelization:

‚úÖ Use parallel: - Large datasets (> 10 GB) - CPU-intensive operations (PCA, SVD) - Multiple cores available - Operations that scale well (matrix mult)

‚ùå Don‚Äôt use parallel: - Small datasets (< 100 MB) - I/O-bound operations - Single core system - Operations with high overhead

Hierarchical Decomposition

For extremely large datasets, use hierarchical processing:

# PCA with hierarchical decomposition
# Divides problem into smaller subproblems
pca <- bdPCA_hdf5(
  filename = "huge_data.hdf5",
  group = "genomics",
  dataset = "snp_matrix",
  k = 20,
  q = 2,        # Hierarchy levels (1-3)
  threads = 8   # Parallel at each level
)

# q = 1: Single level (default)
# q = 2: Two-level hierarchy (recommended for > 100 GB)
# q = 3: Three-level hierarchy (for > 1 TB)

Performance gain:

Single level (q=1):  Handles up to ~100 GB
Two levels (q=2):    Handles up to ~1 TB
Three levels (q=3):  Handles any size

Monitoring Performance

Track performance with simple benchmarking:

# Benchmark operation
system.time({
  result <- bdPCA_hdf5(
    filename = "data.hdf5",
    group = "test",
    dataset = "matrix",
    k = 20,
    threads = 4
  )
})

# Compare different configurations
configs <- data.frame(
  threads = c(1, 2, 4, 8),
  blocksize = c(512, 256, 128, 64)
)

times <- sapply(1:nrow(configs), function(i) {
  system.time({
    bdPCA_hdf5("data.hdf5", "test", "matrix",
               k = 20,
               threads = configs$threads[i])
  })[3]
})

# Find best configuration
best_config <- configs[which.min(times), ]
