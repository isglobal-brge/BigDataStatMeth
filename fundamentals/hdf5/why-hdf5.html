Why HDF5?

Motivation

Working with large matrices in memory (RAM) does not scale. Formats such as CSV or RData require loading entire files, which makes I/O slow and prevents out-of-memory computation.

HDF5 (Hierarchical Data Format v5) stores data on disk with a hierarchical structure and efficient binary encoding. Datasets can be read/written partially (by blocks), enabling scalable algorithms.

Important

Key idea — HDF5 lets BigDataStatMeth operate on matrices that do not fit in memory, using block-wise algorithms implemented in C++.

Why BigDataStatMeth uses HDF5

  • Block-wise computation: data is processed in manageable blocks, minimizing RAM usage.
  • C++ backend: heavy lifting (I/O + math) runs in C++ for speed.
  • Interoperability: the same HDF5 file can be inspected in R (rhdf5 package), C++, Python or MATLAB.
  • Reproducibility: raw data, metadata and results can live in the same file (groups/datasets/attributes).
Note

Note on reading data — In R, there is no general read_* wrapper because computations operate directly on HDF5 files.
If you want to inspect file contents, you can use the functions h5ls() or h5read() from the rhdf5 package.
All internal reads are performed in C++ by BigDataStatMeth.

How HDF5 compares to text/binary formats

Feature CSV / RData HDF5
Data access Whole file Partial / block
File size Large (text) Compact (binary)
Metadata Weak / absent Rich (attributes)
Scalability Limited by RAM Out-of-memory
Cross-language Low High (R/C++/Py/MATLAB)

Mini-benchmark: in-memory vs on-disk

Below we compare X'X using base R (crossprod()), the in-memory function from BigDataStatMeth (bdCrossprod()), and the on-disk HDF5 variant (bdCrossprod_hdf5()).

  • R
  • C++ (not executed)
library(BigDataStatMeth)
library(microbenchmark)

set.seed(1)
X <- matrix(rnorm(5e5), nrow = 1000)

bm <- microbenchmark(
  base_R   = crossprod(X),
  bd_mem   = bdCrossprod(X),
  bd_hdf5  = {
    bdCreate_hdf5_matrix("bench.h5", X, group = "data", dataset = "X", 
                         overwriteFile = TRUE)
    bdCrossprod_hdf5("bench.h5", group = "data", A = "X")
  },
  times = 5
)

The median runtime for bdCrossprod() is 9.19 ms, while base R’s crossprod() takes 145.275 ms (15.807× slower). Using the HDF5-based bdCrossprod_hdf5() requires about 226.207 ms (24.614 × slower), which is expected due to disk I/O. However, this variant enables analysis on datasets far larger than the available RAM.

#include "BigDataStatMeth.h"
int main() {
  // Example: On-disk cross-product using BigDataStatMeth C++ API
  H5Matrix X("bench.h5", "/data/X");
  auto XtX = bdCrossprod_hdf5(X);
  XtX.save("bench_result.h5", "/results/XtX");
  return 0;
}
Tip

How to read results — The HDF5-based function bdCrossprod_hdf5() is naturally slower than in-memory computations, as it performs disk I/O. Yet, it allows operations on matrices that exceed system memory, enabling truly scalable analysis.

Interoperability and Portability in Practice

One of the strongest advantages of using HDF5 is its universal compatibility across both software ecosystems and operating systems. A single file can move seamlessly between environments without any conversion, preserving structure, metadata, and data integrity.

TipOne File, Any Platform

Create on Linux, analyze on macOS, visualize on Windows —
HDF5 files are binary-compatible across all operating systems. Move effortlessly between local workstations, HPC clusters, and cloud infrastructure without reformatting or data loss.

Software Interoperability

The same HDF5 file can be accessed by different programming languages without modification:

TipCross-Language
  • R (BigDataStatMeth, rhdf5)
  • Python (h5py, PyTables)
  • MATLAB (h5read)
  • Julia (HDF5.jl)
  • C/C++ (native HDF5 library)
TipCross-Platform
  • Created on Linux cluster
  • Analyzed on macOS laptop
  • Visualized on Windows desktop
  • Archived in cloud storage
  • Shared with collaborators worldwide

This makes HDF5 a truly universal exchange format for data-driven research: the same file can travel from one system or team to another while keeping every numeric value and attribute intact.

Real-world Scenarios

NoteCollaborative Research Example

Scenario: International multi-site genomic study

  • Site 1 (USA): Genotyping on a Linux cluster → creates HDF5
  • Site 2 (Europe): Quality control on macOS → reads and updates
  • Site 3 (Asia): Statistical analysis on Windows → same HDF5
  • Central Repository: stored in AWS S3 for all sites to access

Key advantage: One data format, one file per dataset, universal compatibility — no conversion steps required.

Benefits

  • Move data effortlessly between computing environments.
  • Upgrade hardware or migrate systems without conversion.
  • Archive data confidently — HDF5 has been stable since 1998.
  • Enable transparent, multi-language, multi-platform workflows.
  • Guarantee reproducibility and traceability across teams.
TipSummary
  • HDF5 enables block-wise, out-of-memory computation.
  • BigDataStatMeth performs I/O and math in C++ for maximum efficiency.
  • Disk-based operations trade a small speed loss for massive scalability.
  • HDF5 ensures universal interoperability and portability, enabling reproducible research across systems and teams.
