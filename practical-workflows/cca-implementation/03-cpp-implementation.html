CCA Implementation in C++ API

Using BigDataStatMeth C++ Header-Only Library

Overview

When to Use C++ vs R

The BigDataStatMeth package provides both R and C++ APIs. Understanding when to use each is crucial:

Use R when: - Rapid prototyping and exploratory analysis - You’re comfortable with R syntax - Integration with other R packages is important - Code readability for collaborators is a priority

Use C++ when: - Building production pipelines - You need full control over memory management - Integrating with existing C++ codebases - Developing new methods that will be packaged

Important Note: Most R functions in BigDataStatMeth are already implemented with C++ backends via Rcpp. This means the performance difference between calling the R function vs the C++ API directly is often minimal. The main advantage of the C++ API is flexibility and control, not raw speed.

What This Document Covers

We’ll implement the same CCA algorithm shown in 02-r-implementation.qmd, but using the C++ header-only API. You’ll see:

  1. How to work with HDF5 datasets using C++ classes
  2. Memory management with pointers (new/delete)
  3. Error handling with try-catch blocks
  4. Calling R functions from C++ when needed

Prerequisites

  • Basic C++ knowledge (classes, pointers, exceptions)
  • Familiarity with the R implementation (see 02-r-implementation.qmd)
  • Understanding of HDF5 file structure

Part 1: Helper Function - getQRbyBlocks_rcpp()

This is the C++ equivalent of the R function getQRbyBlocks() we saw in the previous document.

Complete Code with Comments

#include <Rcpp.h>
#include "BigDataStatMeth.hpp"

void getQRbyBlocks_rcpp(
    hdf5Dataset *dsA,      // Pointer to HDF5 dataset
    int mblocks,           // Number of blocks
    bool bcenter,          // Center data?
    bool bscale,           // Scale data?
    bool byrows,           // Process by rows?
    bool overwrite,        // Overwrite existing results?
    Rcpp::Nullable<int> threads = R_NilValue) 
{
    hdf5Dataset* dstmp = nullptr;  // Temporary dataset pointer
    
    try {
        std::string strInPath, strOutPath;
        bool bycols = !byrows;
        std::string strdataset = dsA->getDatasetName();
        
        // Step 1: Normalize data (center/scale)
        RcppNormalizeHdf5(dsA, bcenter, bscale, byrows);
        
        // Step 2: Open normalized dataset
        dstmp = new hdf5Dataset(
            dsA->getFileName(), 
            "NORMALIZED/" + dsA->getGroupName(), 
            strdataset, 
            false
        );
        dstmp->openDataset();
        
        // Step 3: Split matrix into blocks by rows
        strOutPath = "Step1/" + strdataset + "rows";
        RcppSplit_matrix_hdf5_internal(
            dstmp, strOutPath, strdataset, bycols,
            mblocks, -1, dstmp->nrows(), dstmp->ncols()
        );
        delete dstmp; 
        dstmp = nullptr;
        
        // Step 4: Compute QR for each block
        StringVector blocks = dsA->getDatasetNames(strOutPath, "", "");
        strInPath = strOutPath;
        strOutPath = "Step2/" + strdataset + "rows";
        
        RcppApplyFunctionHdf5(
            dsA->getFileName(), strInPath, blocks, strOutPath, "QR",
            R_NilValue, R_NilValue, wrap(overwrite), 
            wrap(false), wrap(false), wrap(false), wrap(byrows), threads
        );
        
        // Step 5: Merge R matrices from QR decompositions
        StringVector blocks_qr = dsA->getDatasetNames(
            strOutPath, strdataset, ".R"
        );
        strInPath = strOutPath;
        strOutPath = "Step3/merged";
        
        RcppBind_datasets_hdf5(
            dsA->getFileName(), strInPath, blocks_qr,
            strOutPath, strdataset + "Rt", "bindRows", 
            false, wrap(overwrite)
        );
        
        // Step 6: Final QR on merged R matrices
        strInPath = strOutPath;
        strOutPath = "Step3/Final_QR";
        
        RcppApplyFunctionHdf5(
            dsA->getFileName(), strInPath, strdataset + "Rt", 
            strOutPath, "QR",
            R_NilValue, R_NilValue, wrap(overwrite),
            wrap(false), wrap(false), wrap(false), wrap(byrows), threads
        );
        
        // Step 7: Split final Q matrix into blocks
        strInPath = strOutPath;
        strOutPath = "Step4/splitted";
        
        dstmp = new hdf5Dataset(
            dsA->getFileName(), strInPath, 
            strdataset + "Rt.Q", false
        );
        dstmp->openDataset();
        
        RcppSplit_matrix_hdf5_internal(
            dstmp, strOutPath, strdataset + "Rt.Q", bycols,
            mblocks, -1, dstmp->nrows(), dstmp->ncols()
        );
        delete dstmp; 
        dstmp = nullptr;
        
        // Step 8: Multiply block Q matrices with split final Q
        strInPath = "Step2/" + strdataset + "rows";
        strOutPath = "Step5";
        CharacterVector b_group = "Step4/splitted";
        
        blocks_qr = dsA->getDatasetNames(strInPath, strdataset, ".Q");
        StringVector b_blocks = dsA->getDatasetNames(
            "Step4/splitted", strdataset + "Rt.Q", ""
        );
        
        RcppApplyFunctionHdf5(
            dsA->getFileName(), strInPath, blocks_qr, 
            strOutPath, "blockmult",
            b_group, b_blocks, wrap(overwrite), 
            wrap(false), wrap(false), wrap(false), wrap(byrows), threads
        );
        
        // Step 9: Bind all Q matrices into final result
        strInPath = strOutPath;
        strOutPath = "Step6";
        blocks = dsA->getDatasetNames(strInPath, strdataset + ".", "");
        
        RcppBind_datasets_hdf5(
            dsA->getFileName(), strInPath, blocks,
            strOutPath, strdataset + "Q", "bindRows", 
            false, wrap(overwrite)
        );
        
    } catch(std::exception& ex) {
        // Cleanup on error
        checkClose_file(dsA, dstmp);
        Rcpp::Rcout << "c++ exception getQRbyBlocks_rcpp: " 
                    << ex.what() << "\n";
        return void();
    }
    
    return void();
}

Step-by-Step Explanation

Step 1: Normalize Data

RcppNormalizeHdf5(dsA, bcenter, bscale, byrows);

What it does: Centers and/or scales the dataset in-place within the HDF5 file.

Why in C++: Direct access to the normalization function without R overhead.

Where stored: Results go to NORMALIZED/[original_group]/[dataset_name]

Memory management: The function handles its own memory; we don’t need to worry about cleanup here.

Step 2: Open Normalized Dataset

dstmp = new hdf5Dataset(
    dsA->getFileName(), 
    "NORMALIZED/" + dsA->getGroupName(), 
    strdataset, 
    false  // false = open existing dataset
);
dstmp->openDataset();

What it does: Creates a C++ object (hdf5Dataset) that wraps the HDF5 dataset.

C++ specifics: - new allocates memory on the heap - We use a pointer because we’ll delete it later - false parameter means “open existing” (not create new) - We must call openDataset() explicitly to actually open it

Memory safety: Notice we initialize dstmp = nullptr at the start. This is crucial for the try-catch cleanup.

Step 3: Split Matrix into Blocks

strOutPath = "Step1/" + strdataset + "rows";
RcppSplit_matrix_hdf5_internal(
    dstmp, strOutPath, strdataset, bycols,
    mblocks, -1, dstmp->nrows(), dstmp->ncols()
);
delete dstmp; 
dstmp = nullptr;

What it does: Splits the normalized matrix into mblocks blocks (by rows or columns).

Why delete here: We’re done with dstmp for now, so we free the memory. Setting to nullptr prevents accidental reuse.

Where stored: Creates datasets named like: - Step1/Xrows/X.0001 - Step1/Xrows/X.0002 - … up to X.mblocks

C++ difference from R: In R, garbage collection handles cleanup automatically. In C++, we must explicitly delete.

Step 4-6: QR Decomposition Pattern

This follows the same block-wise QR algorithm as the R version:

  1. Step 4: Compute QR for each block separately
  2. Step 5: Merge all R matrices vertically
  3. Step 6: Compute final QR on the merged R matrix

C++ advantages here: - Direct function calls without R interpreter overhead - More control over memory allocation for intermediate results - Can optimize memory reuse (though this implementation doesn’t yet)

Key C++ function:

RcppApplyFunctionHdf5(
    dsA->getFileName(),    // File to work on
    strInPath,             // Input group
    blocks,                // Which datasets to process
    strOutPath,            // Output group
    "QR",                  // Function to apply
    ...                    // Additional parameters
);

This applies the QR decomposition to multiple datasets in parallel.

Step 7-9: Reconstruct Full Q Matrix

Similar pattern to Steps 4-6, but working with Q matrices:

  1. Step 7: Split the final Q matrix into blocks
  2. Step 8: Multiply block Q matrices with corresponding pieces of final Q
  3. Step 9: Bind results vertically to get complete Q matrix

Why this complexity: The block-wise algorithm means we have Q matrices at two levels: - Block-level Q matrices (from Step 4) - Final Q matrix (from Step 6)

To get the true Q matrix of the original data, we need to multiply these appropriately.

Error Handling

} catch(std::exception& ex) {
    checkClose_file(dsA, dstmp);
    Rcpp::Rcout << "c++ exception getQRbyBlocks_rcpp: " 
                << ex.what() << "\n";
    return void();
}

What it does: If any exception occurs, we: 1. Clean up any open file handles 2. Print the error message 3. Return gracefully (not crash)

Why necessary in C++: Unlike R, C++ doesn’t automatically clean up resources on error. We must do it explicitly.

Best practice: Always use try-catch when working with resources (files, memory) in C++.

Part 2: Main CCA Function - bdCCA_hdf5_rcpp()

Now let’s look at the main CCA function that orchestrates everything.

Complete Code

// [[Rcpp::export]]
void bdCCA_hdf5_rcpp(
    std::string filename, 
    std::string datasetX,
    std::string datasetY, 
    bool bcenter, 
    bool bscale, 
    int mblocks,
    bool overwrite, 
    Rcpp::Nullable<int> threads = R_NilValue) 
{
    // Initialize all pointers to nullptr for safety
    hdf5Dataset* dsX = nullptr;
    hdf5Dataset* dsY = nullptr;
    hdf5Dataset* dsXQ = nullptr;
    hdf5Dataset* dsYQ = nullptr;
    hdf5Dataset* dsC = nullptr;

    try {
        int ncolsX, ncolsY;

        // Step 1-6: QR decomposition of X
        dsX = new hdf5Dataset(filename, datasetX, false); 
        dsX->openDataset(); 
        
        getQRbyBlocks_rcpp(
            dsX, mblocks, bcenter, bscale, 
            false, overwrite, threads
        );
        
        // Step 1-6: QR decomposition of Y
        dsY = new hdf5Dataset(filename, datasetY, false); 
        dsY->openDataset(); 
        
        getQRbyBlocks_rcpp(
            dsY, mblocks, bcenter, bscale, 
            false, overwrite, threads
        );

        // Step 7: Crossproduct of Q matrices
        dsXQ = new hdf5Dataset(filename, "Step6", "XQ", false); 
        dsXQ->openDataset(); 
        
        dsYQ = new hdf5Dataset(filename, "Step6", "YQ", false); 
        dsYQ->openDataset(); 
        
        dsC = new hdf5Dataset(filename, "Step7", "CrossProd_XQ_YQ", overwrite);
        
        // Compute optimal block size for multiplication
        int optimBlock = getMaxBlockSize(
            dsXQ->nrows(), dsXQ->ncols(), 
            dsYQ->nrows(), dsYQ->ncols(), 
            2, R_NilValue
        );
        
        // Perform crossproduct: t(XQ) %*% YQ
        dsC = BigDataStatMeth::crossprod(
            dsXQ, dsYQ, dsC, 
            optimBlock, optimBlock/2, 
            true, true, threads
        );
        
        // Clean up intermediate datasets
        delete dsXQ; dsXQ = nullptr;
        delete dsYQ; dsYQ = nullptr;
        
        // Save dimensions for later use
        ncolsX = dsX->ncols_r();
        ncolsY = dsY->ncols_r();
        
        // Step 8: SVD of crossproduct
        RcppbdSVD_hdf5(
            dsC->getFileName(), 
            dsC->getGroupName(), 
            dsC->getDatasetName(),  
            16,           // k = 16 components
            2,            // q = 2 levels
            0,            // p = 0 (default)
            false,        // center = false (already done)
            false,        // scale = false
            0,            // method = 0 (default)
            overwrite, 
            false,        // irlba = false
            R_NilValue,   // randstart
            R_NilValue,   // threads
        );
        
        // Clean up
        delete dsC; dsC = nullptr;
        delete dsX; dsX = nullptr;
        delete dsY; dsY = nullptr;
        
        // Step 9: Compute CCA components (call R function)
        writeCCAComponents_hdf5_rcpp(filename, ncolsX, ncolsY);
        
    } catch(std::exception& ex) {
        // Cleanup all pointers on error
        checkClose_file(dsX, dsY, dsXQ, dsYQ, dsC);
        Rcpp::Rcerr << "\nc++ exception bdCCA_hdf5_rcpp: " 
                    << ex.what() << "\n";
        return void();
    }

    return void();
}

Step-by-Step Explanation

Initialization

hdf5Dataset* dsX = nullptr;
hdf5Dataset* dsY = nullptr;
hdf5Dataset* dsXQ = nullptr;
hdf5Dataset* dsYQ = nullptr;
hdf5Dataset* dsC = nullptr;

Why initialize to nullptr: If an exception occurs early, the catch block tries to clean up all pointers. If they’re not initialized, we might try to delete garbage memory → crash.

Best practice: Always initialize pointers, especially when using try-catch.

Steps 1-6: QR for X and Y

dsX = new hdf5Dataset(filename, datasetX, false); 
dsX->openDataset(); 

getQRbyBlocks_rcpp(
    dsX, mblocks, bcenter, bscale, 
    false, overwrite, threads
);

What it does: Opens dataset X and computes its block-wise QR decomposition.

Parameter notes: - false in hdf5Dataset constructor = open existing dataset - false in getQRbyBlocks_rcpp = process by columns (not rows) - We pass the dataset pointer directly (no copying)

Memory: The hdf5Dataset object is lightweight (just metadata), not the actual matrix data.

Step 7: Crossproduct

int optimBlock = getMaxBlockSize(
    dsXQ->nrows(), dsXQ->ncols(), 
    dsYQ->nrows(), dsYQ->ncols(), 
    2, R_NilValue
);

dsC = BigDataStatMeth::crossprod(
    dsXQ, dsYQ, dsC, 
    optimBlock, optimBlock/2, 
    true, true, threads
);

What it does: Computes C = t(X_Q) × Y_Q using optimal block size.

Why getMaxBlockSize: Automatically determines the best block size based on: - Matrix dimensions - Available memory - Cache size considerations

C++ advantage: Direct access to low-level optimization parameters.

Where stored: Step7/CrossProd_XQ_YQ

Step 8: SVD

RcppbdSVD_hdf5(
    dsC->getFileName(), 
    dsC->getGroupName(), 
    dsC->getDatasetName(),  
    16,    // k = 16 components
    2,     // q = 2 levels
    ...
);

What it does: Computes hierarchical SVD of the crossproduct matrix.

Parameters explained: - k = 16: Compute first 16 singular values (canonical correlations) - q = 2: Use 2-level hierarchy for large matrices - center/scale = false: Data already preprocessed

Result: Singular values (correlations), left singular vectors (u), right singular vectors (v)

Where stored: SVD/CrossProd_XQ_YQ/d, u, v

Step 9: Extract Components

ncolsX = dsX->ncols_r();
ncolsY = dsY->ncols_r();

writeCCAComponents_hdf5_rcpp(filename, ncolsX, ncolsY);

What it does: Calls an R function from C++ to compute the final CCA coefficients and scores.

Why call R here: The coefficient computation involves matrix operations that are already well-optimized in R, so there’s little benefit to rewriting in C++.

How it works:

void writeCCAComponents_hdf5_rcpp(
    std::string filename, int ncolsX, int ncolsY)
{
    Rcpp::Environment base("package:BDStatMethExamples"); 
    Rcpp::Function write_components = base["writeCCAComponents_hdf5"];    
    
    write_components(
        Rcpp::_["filename"] = wrap(filename),
        Rcpp::_["ncolsX"]  = ncolsX,
        Rcpp::_["ncolsY"]  = ncolsY
    ); 
}

Key points: - Environment loads the R package - Function gets the R function by name - We can call it like a C++ function with named parameters

Pragmatic approach: Use C++ where it helps (file I/O, memory control), use R where it’s convenient (high-level operations).

Error Handling and Cleanup

} catch(std::exception& ex) {
    checkClose_file(dsX, dsY, dsXQ, dsYQ, dsC);
    Rcpp::Rcerr << "\nc++ exception bdCCA_hdf5_rcpp: " 
                << ex.what() << "\n";
    return void();
}

What checkClose_file does: For each non-null pointer: 1. Close HDF5 file handle if open 2. Delete the C++ object 3. Set pointer to nullptr

Why multiple pointers: We might have opened several datasets before an error occurs. We need to close all of them.

Error reporting: We use Rcerr (not Rcout) to print errors to stderr.

Using from R

Once compiled, you can call it from R:

Verification Example

Let’s verify the C++ version gives the same results as the R version:

library(BigDataStatMeth)
library(rhdf5)

# Generate test data
set.seed(123)
n <- 1000
X <- matrix(rnorm(n * 50), n, 50)
Y <- matrix(rnorm(n * 30), n, 30)

# Create two separate files for comparison
bdCreate_hdf5_matrix("cca_r.h5", X, "data", "X", overwriteFile = TRUE)
bdCreate_hdf5_matrix("cca_r.h5", Y, "data", "Y")

bdCreate_hdf5_matrix("cca_cpp.h5", X, "data", "X", overwriteFile = TRUE)
bdCreate_hdf5_matrix("cca_cpp.h5", Y, "data", "Y")

# Run both versions
bdCCA_hdf5("cca_r.h5", "data/X", "data/Y", m = 4)  # R version
bdCCA_hdf5_rcpp("cca_cpp.h5", "data/X", "data/Y",  # C++ version
                bcenter = TRUE, bscale = FALSE, 
                mblocks = 4, overwrite = TRUE)

# Compare results
cor_r <- h5read("cca_r.h5", "/Results/cor")
cor_cpp <- h5read("cca_cpp.h5", "/Results/cor")

all.equal(cor_r, cor_cpp, tolerance = 1e-10)
# [1] TRUE

Expected result: Both versions should give identical results (within numerical precision).

Key Differences: R vs C++

Memory Management

R:

# Automatic garbage collection
result <- some_function(data)
# 'data' automatically cleaned up when no longer needed

C++:

// Manual memory management
hdf5Dataset* ds = new hdf5Dataset(filename, dataset, false);
// ... use ds ...
delete ds;  // Must explicitly free memory
ds = nullptr;  // Good practice

Error Handling

R:

tryCatch({
  result <- risky_operation()
}, error = function(e) {
  message("Error: ", e$message)
})
# Resources automatically cleaned up

C++:

try {
  result = risky_operation();
} catch(std::exception& ex) {
  // Must manually clean up resources
  cleanup_resources();
  Rcpp::Rcout << "Error: " << ex.what() << "\n";
}

Function Calls

R:

# High-level, flexible
result <- bdCrossprod_hdf5(
  filename = "file.h5",
  group = "Step6",
  A = "XQ",
  groupB = "Step6",
  B = "YQ"
)

C++:

// More explicit, more control
hdf5Dataset* dsXQ = new hdf5Dataset(filename, "Step6", "XQ", false);
hdf5Dataset* dsYQ = new hdf5Dataset(filename, "Step6", "YQ", false);
hdf5Dataset* dsC = new hdf5Dataset(filename, "Step7", "result", true);

dsC = BigDataStatMeth::crossprod(
  dsXQ, dsYQ, dsC, 
  blocksize, blocksize/2,
  true, true, threads
);

When to Choose Each

Choose R implementation when: - You need results quickly (prototyping) - Code clarity is more important than control - You’re working interactively - Your collaborators aren’t C++ programmers

Choose C++ implementation when: - Building a production pipeline - You need to integrate with other C++ code - You want to customize low-level behavior - You’re developing a new method that will be packaged

Realistic expectation: For most use cases with BigDataStatMeth, the R functions (which already use C++ backends) will be sufficient. The C++ API is for advanced users who need fine-grained control.
